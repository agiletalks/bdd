BDD 需求結構與階層關係教學

理解 Feature ── User Story ── Scenario ── Test Case ── GWT ── Playwright Code 的層次關係

在 BDD 與敏捷開發流程中，需求是從「大且抽象的系統功能」逐步細化為與 AI 協作的「小且具體的可執行規格」。以下是這五個需求層級的結構與階層關係：

Feature 功能特性

視角：系統層面 (What) 產出角色：Product Owner (PO) / 系統分析 (SA)

系統要提供的一個功能模組或大型服務項目（通常是系統名稱或模組名稱）。

典型範例：線上非約定轉帳
2

User Story 使用者故事

視角：使用者價值面 (Who & Why) 產出角色：PO / 業務分析 (BA)

從特定使用者角色的角度，描述他希望在此功能中完成什麼事，以及這個動作帶給他的業務價值。

標準格式：身為 [使用角色]，我想要 [進行何種操作]，以便於 [達成何種商業價值]
故事範例：身為一個 行動銀行 App 用戶，我想要 在線上轉帳給朋友，以便於 即時處理資金需求。
3

Scenario 業務情境

視角：業務流程路徑 (Which Flow) 產出角色：SA / BA / 測試人員 (QA)

為了滿足同一個 User Story，在不同分支、條件下的完整業務流程路徑。必須是一個端到端 (End-to-End) 的完整情境。

情境範例：
  • 情境 A：帳戶餘額充足，轉帳成功 (正常路徑)
  • 情境 B：轉帳金額超過單筆限額，轉帳失敗 (業務限制)
  • 情境 C：收款銀行系統逾時，轉帳失敗 (例外路徑)
4

Test Case 測試案例/條件

視角：測試與防呆面 (How to verify) 產出角色：QA / 測試分析師

針對每個 Scenario 下，去驗證系統輸入、極端值、防呆條件的具體可驗證條件。

表格化測試案例範例（以情境 B「超過單筆限額」為例）：
Feature ID User Story ID Scenario ID Test Case ID Test Case 名稱 測試目的 測試類型
F01 (轉帳) US01 (線上轉帳) S02 (超過限額) TC01-01 轉帳金額等於單筆限額 驗證單筆最高限額邊界值是否可正常扣款與轉帳 邊界案例
F01 (轉帳) US01 (線上轉帳) S02 (超過限額) TC01-02 轉帳金額大於單筆限額 驗證金額超過單筆限額時系統應成功攔截並報錯 例外案例
F01 (轉帳) US01 (線上轉帳) S02 (超過限額) TC01-03 轉帳金額輸入負數 防呆驗證，確保轉帳金額輸入負數時系統無法提交 欄位驗證
5

Given - When - Then 行為規格步驟

視角：精確行為規格 (Executable Spec) 產出角色：BA / QA / 開發團隊 (Dev)

將測試案例用無歧義的結構化語言（Gherkin 語法）撰寫成可被開發與自動化測試執行的行為規格。

標準結構：
Given：前提條件（角色身分、帳戶餘額、頁面狀態）
When：觸發動作（使用者輸入、點擊按鈕）
Then：預期結果（畫面跳轉、錯誤提示、扣款狀態）
表格化 Given-When-Then 範例（對應上述 TC01-02 測試案例）：
對應 Test Case Given 前提條件 When 觸發動作 Then 預期結果 Expected Value 預期數值/狀態
TC01-02
(轉帳金額大於單筆限額)		 1. 用戶登入成功
2. 帳戶可用餘額有 $100,000
3. 系統單筆轉帳上限為 $50,000 		1. 在轉帳金額輸入 $60,000
2. 點擊「確認轉帳」按鈕 		1. 系統阻擋交易送出
2. 頁面不進行跳轉
3. 轉帳金額欄位下方顯示紅字錯誤提示 		錯誤提示文字為：
轉帳金額已超過單筆限額 $50,000 元

Playwright Automated Code 自動化測試腳本

視角：測試代碼與執行 (E2E Test) 產出角色：Dev / QA 自動化測試工程師

將 Given-When-Then 表格中的語意化動作與預期值直接轉化為可由瀏覽器自動執行與驗證的程式碼。使專案能夠一鍵自動回歸，確保後續功能迭代時不影響既有業務邏輯。

Playwright 自動化測試代碼範例（對應上述 TC01-02 測試條件）： 
import { test, expect } from '@playwright/test';

test('TC01-02 轉帳金額大於單筆限額應阻擋並報錯', async ({ page }) => {
  // Given: 用戶登入成功，且帳戶可用餘額有 $100,000，單筆限額為 $50,000
  await page.goto('/login');
  await page.fill('#username', 'testuser');
  await page.click('#login-submit-btn');
  
  // When: 在轉帳金額輸入 $60,000，並點擊「確認轉帳」按鈕
  await page.fill('#transfer-amount', '60000');
  await page.click('#confirm-transfer-btn');
  
  // Then: 系統阻擋交易送出，轉帳金額欄位下方顯示錯誤提示
  const errorAlert = page.locator('.error-message');
  await expect(errorAlert).toBeVisible();
  await expect(errorAlert).toHaveText('轉帳金額已超過單筆限額 $50,000 元');
});

步驟 1：語音需求訪談助理產出

成果階層：非結構化訪談摘要 產出角色：引導式 BA 角色 (Prompt 0)

AI 防插話並引導出業務目標、故事主線與核心限制，訪談結束後一鍵生成結構化摘要：

Prompt 0 訪談結束彙整報告：
  • 訪談背景：提供行動銀行用戶快速線上轉帳功能，處理即時資金需求。
  • 核心故事 (Golden Path)：用戶在轉帳頁面輸入收款帳號與轉帳金額，點擊送出，驗證成功後直接完成扣款與轉帳。
  • 業務規則與例外防呆：
    • 單筆轉帳上限為新台幣 $50,000 元。
    • 若轉帳金額大於單筆上限（$50,000），阻擋送出並顯示提示。

步驟 5：Playwright E2E 自動化測試代碼

成果階層：可執行測試腳本 (.spec.ts) 產出角色：自動化 QA 工程師角色 (Prompt 4)

AI 讀取 GWT 規格表格，直接翻譯為可執行的 Playwright TypeScript 代碼，打通開發與測試閉環：

Playwright 代碼產出（對應上述 TC01-02 流程）： 
import { test, expect } from '@playwright/test';

test.describe('行動銀行轉帳限額限制驗證', () => {

  test('TC01-02 轉帳金額大於單筆限額應阻擋並報錯', async ({ page }) => {
    // Given: 用戶登入帳戶成功，且可用餘額有 $100,000，單筆限額為 $50,000
    await page.goto('/login');
    await page.fill('#username', 'testuser123');
    await page.click('#btn-login-submit');
    
    // When: 在轉帳金額輸入欄位填入 60000，並點選「確認轉帳」按鈕
    await page.fill('#transfer-amount', '60000');
    await page.click('#confirm-transfer-btn');
    
    // Then: 系統阻擋交易送出，頁面不進行跳轉，且轉帳金額欄位下方顯示錯誤提示
    const errorAlert = page.locator('.error-message');
    await expect(errorAlert).toBeVisible();
    await expect(errorAlert).toHaveText('轉帳金額已超過單筆限額 $50,000 元');
  });

});
